# Contributing to Pellis

Thanks for being here this early. Right now Pellis is foundation — vision,
governance, licensing — so the most valuable contributions are about **direction and
shape**, not yet code (there isn't any to write against yet).

## What's useful right now

- **Honest reactions to the direction.** Where the strategy is wrong, where it's
  right, where you'd push harder or start differently. Open an issue.
- **Domain expertise.** Soft/stretchable PCBs, conductive textiles, on-device ML,
  haptics, washable power, garment manufacturing — if you know one of these areas,
  your read on what's tractable is gold.
- **Prior art and references.** Papers, projects, parts, or teams doing adjacent work.

## When there's code and hardware to contribute

Once the device library and hardware spec land, the flow will be the standard one:

1. Open an issue to discuss the change before you build it, so effort isn't wasted.
2. Fork, branch, and make the change.
3. Open a pull request that explains *what* and *why*.
4. A maintainer for the relevant layer reviews and merges.

Every contributor is credited in [`CONTRIBUTORS.md`](CONTRIBUTORS.md). How roles work —
collaborator, maintainer, project lead — is described in
[`GOVERNANCE.md`](GOVERNANCE.md). Everyone starts as a collaborator; you earn
maintainer status by owning a layer.

## Licensing of contributions

By contributing, you agree your contribution is licensed under the license that
applies to that artifact type, per [`LICENSING.md`](LICENSING.md) (CERN-OHL-S v2 for
hardware, Apache-2.0 for firmware/library/models, CC-BY for docs).

## Ground rules

Be the kind of collaborator you'd want to work with. The baseline is in
[`CODE_OF_CONDUCT.md`](CODE_OF_CONDUCT.md).
